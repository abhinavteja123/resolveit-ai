## 1. Project Overview

**ResolveIT AI** is a full-stack RAG (Retrieval-Augmented Generation) powered IT support assistant. It allows IT teams to upload runbooks (PDF, DOCX, TXT) and then query them in natural language to get step-by-step resolutions generated by Google Gemini, with citations back to the source documents.

### Who uses it

| Role | What they do |
|---|---|
| **IT Engineer** | Asks questions like "how do I fix Apache 502?" and gets instant, cited answers |
| **Admin** | Uploads and manages the shared knowledge base (runbooks) |
| **Any authenticated user** | Can upload their own private runbooks and query across scopes |

### Core capabilities

- Multi-format runbook ingestion (PDF, DOCX, TXT)
- Hybrid semantic + keyword search with cross-encoder re-ranking
- Real-time streaming answers via Server-Sent Events (SSE)
- Confidence scoring and source attribution on every answer
- Thumbs-up/down feedback loop with admin analytics dashboard
- Scoped retrieval: admin runbooks, personal runbooks, or both
- Full query audit log with pagination

## 5. Data Flow

### Query flow (happy path)

```
User types question in Dashboard
        │
        ▼
useQuery hook → POST /query/stream (Bearer token, scope)
        │
        ▼  [Backend]
1. Firebase JWT verified
2. HyDE expansion
   └─ Gemini generates a hypothetical runbook passage for the query
   └─ Combined: original_query + hypothetical_passage
3. Hybrid retrieval
   ├─ BGE embedder encodes expanded query
   ├─ FAISS → top 12 semantic matches
   ├─ BM25 → top 12 keyword matches
   └─ RRF fusion → ranked candidate list
4. Scope filter
   └─ Keep only chunks matching requested scope (admin / mine / both)
5. Cross-encoder re-ranking
   └─ BGE reranker scores each (query, chunk) pair
   └─ Sigmoid normalize → confidence 0–1
   └─ Top 5 selected
6. Gemini generation
   └─ System prompt: cite inline [1][2], numbered steps, prevention tips
   └─ Stream tokens back via SSE
7. Log to Supabase (query_logs)
8. Cache result (TTLCache, 600s)
        │
        ▼  [Frontend]
SSE: "sources" event → show source badges
SSE: "token" events → stream answer into ResultCard (rendered as Markdown)
SSE: "done" event → show feedback buttons
```

### Ingestion flow

```
Admin uploads file → POST /admin/upload
        │
        ▼
1. Magic-byte validation (reject mismatched type)
2. parser.py → extract raw text
3. chunker.py → recursive split (chunk_size=800, overlap=100)
4. embedder.py → embed all chunks (batched)
5. faiss_store.py → add vectors to FAISS, persist to disk
6. bm25_store.py → add tokens to BM25 (in-memory rebuild)
7. Supabase → insert row in runbooks table
```

## 6. AI / ML Pipeline

### 6.1 Embedding

- **Model**: `BAAI/bge-small-en-v1.5` (384-dimensional dense vectors)
- **Library**: `sentence-transformers`
- **Normalization**: L2-normalize all vectors so inner product = cosine similarity
- **Query prefix**: `"Represent this sentence for searching relevant passages: "` prepended at query time (not at indexing time)
- **Warm-up**: loaded once at server startup via FastAPI `lifespan`

### 6.2 Retrieval — FAISS

- **Index type**: `IndexFlatIP` (exact inner product; cosine on normalized vectors)
- **Storage**: `faiss_index/index.bin` (vectors) + `faiss_index/metadata.json` (chunk metadata)
- **Top-k**: 12 candidates per query
- **Persistence**: saved to disk after every add/delete

### 6.3 Retrieval — BM25

- **Implementation**: `rank-bm25` (`BM25Okapi`)
- **Storage**: in-memory only (rebuilt from metadata on server start)
- **Top-k**: 12 candidates per query
- **Tokenization**: whitespace split

### 6.4 Hybrid Fusion — RRF

```
score(chunk) = Σ  1 / (k + rank_in_list)
               lists

k = 60  (standard RRF smoothing constant)
```

Produces a single ranked list combining dense and lexical signals.

### 6.5 HyDE (Hypothetical Document Expansion)

Before retrieval, Gemini generates a 3–5 sentence hypothetical runbook passage answering the query. Both the original query and this passage are embedded, then the resulting vectors are averaged (or concatenated) for retrieval. This improves recall for questions that don't directly match runbook phrasing.

### 6.6 Cross-Encoder Re-Ranking

- **Model**: `BAAI/bge-reranker-base` (~120M params)
- **Input**: (query, chunk_text) pairs
- **Output**: raw logit → sigmoid → confidence score ∈ [0, 1]
- **Top-k**: top 5 selected after re-ranking
- These top 5 become the context passed to Gemini

### 6.7 LLM Generation (Gemini)

- **Model**: `gemini-2.5-flash` (configurable)
- **Temperature**: 0.2 (deterministic)
- **System prompt** enforces:
  - Inline citation format: `[1]`, `[2]`
  - Structure: **Problem Summary** → numbered **Resolution Steps** → **Prevention Tips**
  - Fallback: "No relevant information found in indexed runbooks" if context is empty
- **Blocking** (`POST /query`): returns full answer at once
- **Streaming** (`POST /query/stream`): SSE events — `sources`, `token` (per chunk), `done`

## 8. Database Schema

### `runbooks`
```sql
CREATE TABLE runbooks (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  filename          TEXT NOT NULL,
  title             TEXT,
  category          TEXT,           -- server | network | application | other
  file_type         TEXT,           -- pdf | docx | txt
  uploaded_by       TEXT,           -- Firebase UID
  uploaded_at       TIMESTAMPTZ DEFAULT NOW(),
  chunk_count       INTEGER,
  is_indexed        BOOLEAN DEFAULT FALSE,
  content_hash      TEXT,           -- SHA256 for deduplication
  is_admin_runbook  BOOLEAN DEFAULT FALSE
);
```

### `query_logs`
```sql
CREATE TABLE query_logs (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           TEXT,           -- Firebase UID
  query_text        TEXT NOT NULL,
  retrieved_sources TEXT[],         -- array of filenames
  llm_response      TEXT,
  confidence_score  FLOAT,
  queried_at        TIMESTAMPTZ DEFAULT NOW()
);
```

### `feedback`
```sql
CREATE TABLE feedback (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  query_log_id    UUID REFERENCES query_logs(id),
  user_id         TEXT,
  rating          INTEGER CHECK (rating IN (1, -1)),  -- 1=up, -1=down
  comment         TEXT,
  submitted_at    TIMESTAMPTZ DEFAULT NOW()
);
```

```
User clicks "Sign in with Google"
        │
        ▼

        │
        ▼
Firebase returns ID token (JWT, ~1h TTL)
        │
        ▼
AuthContext stores token, sets up auto-refresh (every 50 min)
        │

        │
        ▼  [Backend]
```

### Production considerations

- Host frontend build (`npm run build` → `dist/`) on a CDN or Nginx

- FAISS index is local to the backend container — use a persistent volume in production
- For scale-out: move FAISS to a shared store (Pinecone, Weaviate) or replicate index to each instance

## 13. Key Design Decisions

| Decision | Rationale |
|---|---|
| **HyDE expansion** | Dense retrieval fails when query wording doesn't match runbook phrasing; generating a hypothetical passage bridges the vocabulary gap |
| **Hybrid RRF fusion** | Dense search excels at semantic similarity; BM25 catches exact command/error strings; RRF combines both without needing to tune weights |
| **Cross-encoder re-ranking** | Bi-encoder (FAISS) is fast but approximate; cross-encoder reads the full (query, chunk) pair and is significantly more precise, worth the extra latency for top-5 selection |
| **In-memory BM25** | BM25 has no standard binary format; simpler to rebuild from metadata.json on startup than serialize/deserialize |
| **FAISS on disk** | Keeps the index persistent across restarts without a separate vector DB service |
| **In-process TTL cache** | Avoids re-calling Gemini for duplicate queries (common in IT support scenarios) without adding Redis dependency |
| **SSE streaming** | Shows answer incrementally — LLM generation takes 2–5s; streaming keeps the UI responsive |
| **Firebase for auth** | Offloads token management, OAuth, and session security to a managed service |
| **Supabase for data** | Managed Postgres with REST API — no need to run a separate DB server |
| **Admin email whitelist** | Simpler than a database role system for a small team; easily changed via env var |
| **Magic-byte validation** | Prevents malicious file type spoofing (e.g., executable renamed to .pdf) |
| **Scope filtering** | Lets admins keep sensitive runbooks separate from user-uploaded content while still allowing combined queries |
